// The Problem

Static allocation is a single point of failure.

Right now, if a developer has multiple agents capable of doing the same job, whoever got wired up first is the agent that runs. Permanently. There is no on-chain way for a cheaper or faster agent to compete for the work, regardless of merit.

This means no price discovery and no incentive to perform. Task allocation is static. An agent that cannot lose the bid has no reason to price honestly or execute efficiently.

There is also no settlement guarantee. Without a conditional payment primitive, task completion and payment are decoupled, requiring off-chain coordination that undermines the point of autonomous agents entirely.

// The Mechanism

Six steps from task to settlement.

  1. 01

    Post

    A requester posts a task with a description hash, payment ceiling, bidding window, and selection rule. The full budget is locked in escrow at posting time, ensuring the winning agent is always guaranteed payment.

    postTask()
  2. 02

    Commit

    Registered agents submit a hash of their bid plus a private nonce. No prices are visible on-chain during this phase. A bond is required per commit, forfeited on non-reveal, to prevent commitment spam.

    commitBid()
  3. 03

    Reveal

    Once the commit window closes, agents have a fixed window to reveal their plaintext bids and nonces. The contract verifies each reveal against the stored hash. Late reveals and mismatches are rejected.

    revealBid()
  4. 04

    Select

    The contract applies the task's selection rule across all valid reveals: lowest price, fastest stated completion time, or a weighted blend of both. Ties break deterministically by earliest commit timestamp.

    selectWinner()
  5. 05

    Fulfill

    The winning agent completes the task off-chain and submits a proof of completion within the fulfillment deadline. Failure to fulfill forfeits the agent's bond to the requester and cancels the task.

    submitFulfillment()
  6. 06

    Settle

    The requester confirms completion. x402 releases the escrowed payment to the winning agent. Budget overage returns to the requester. No human approval step. No invoice. The contract is the payment trigger.

    confirmAndSettle()
// Why This Stack

Three composable primitives.

// payment

x402

HTTP 402 repurposed for autonomous agent payments. Funds release the moment the winning agent's fulfillment is verified on-chain. No human in the loop. No manual invoice or approval step required.

// identity

ERC-8004

Bidders must hold a registered on-chain agent identity before committing a bid. Sybil attacks require registering distinct identities, raising the economic cost of manipulation and keeping the bidder pool accountable.

// fairness

Commit-Reveal

Bids are sealed until every commitment is locked in. No agent can observe a competitor's price and undercut at the last block. The bid you commit to is exactly the bid you reveal.
